# Git упатство за тимот на Srednoskolski-glas

## Цел на ова упатство

Ова упатство објаснува како тимот треба да работи со Git во овој репозиториум. Проектот е организиран како еден репо со две главни области:

- `backend/` - Laravel апликацијата
- `frontend/` - Next.js апликацијата

Git работи на ниво на целото репо, не на ниво на поединечна папка. Тоа значи дека не постои команда со која ќе "пушнете само frontend" како посебен Git push ако сѐ уште сте во исто репо. Наместо тоа, тимот работи преку јасни правила за гранки, папки и merge процес.

## Основни правила

1. Секој девелопер работи на своја посебна гранка.
2. Backend тимот менува само датотеки во `backend/`, освен ако PM не одобри промена во shared/root датотеки.
3. Frontend тимот менува само датотеки во `frontend/`, освен ако PM не одобри промена во shared/root датотеки.
4. Никој не прави директен commit на `main` гранка.
5. Кога кодот е стабилен и тестиран, девелоперот ја пушта својата гранка и му јавува на PM.
6. PM ги прегледува промените и прави merge во главната гранка.
7. Ако постои конфликт меѓу backend и frontend промени, PM одлучува како се решава.

## Како да се размислува за репото

Ова репо е еден проект со две функционални зони:

- Backend тимот работи во Laravel делот.
- Frontend тимот работи во Next.js делот.

Тоа значи:

- Frontend тимот не смее да менува backend логика, модели, контролери, миграции или API route-ови без договор.
- Backend тимот не смее да менува frontend компоненти, страници или UI логика без договор.
- Ако треба да се менува нешто што ги допира и двата дела, тоа се прави координирано и по договор со PM.

## Препорачана структура на гранки

Секој девелопер работи во формат како:

- `backend/feature-login`
- `backend/fix-auth`
- `frontend/homepage-redesign`
- `frontend/profile-page`
- `dev/name-task-description`

Добра практика е името на гранката да кажува:

- кој дел е засегнат
- што се менува
- кој е авторот или задачата

Пример:

- `frontend/ivan-login-page`
- `backend/amina-api-users`
- `backend/fix-token-refresh`

## Дневен workflow за секој девелопер

### 1. Освежи ја локалната копија

Пред да почнеш, провери дали си на точната гранка и симни ги последните промени.

```bash
git status
git fetch origin
```

Што прават овие команди:

- `git status` - покажува на која гранка си и дали имаш локални промени.
- `git fetch origin` - ги симнува новите промени од remote репото без да ги спојува во твојата гранка.

Ако треба да ја освежиш твојата гранка со најновото од главната гранка:

```bash
git pull origin main
```

Што прави:

- `git pull origin main` - ги симнува и веднаш ги спојува промените од `main` во твојата моментална гранка.


### 2. Направи нова гранка за задачата

Ако почнуваш нова работа:

```bash
git checkout main
git pull origin main
git checkout -b frontend/ivan-login-page
```

Што прават овие команди:

- `git checkout main` - те носи на главната гранка.
- `git pull origin main` - ја ажурира главната гранка локално.
- `git checkout -b ...` - креира нова гранка и веднаш те префрла на неа.

Важно:

- Backend девелопер отвора гранка во backend стил.
- Frontend девелопер отвора гранка во frontend стил.
- Не се работи директно на `main`.

### 3. Работи само во твојата зона

#### Ако си backend девелопер

Работи во:

- `backend/`

Типични backend промени:

- контролери
- модели
- миграции
- API routes
- validation
- auth логика
- business логика

Не менувај:

- `frontend/` освен ако имаш договор
- frontend UI компоненти
- Next.js страници

#### Ако си frontend девелопер

Работи во:

- `frontend/`

Типични frontend промени:

- страници
- компоненти
- styling
- форма логика во UI
- client-side state
- API повици од frontend

Не менувај:

- `backend/` освен ако имаш договор
- Laravel контролери
- backend миграции
- backend auth или API структура

### 4. Провери што си сменил

Повеќе пати во текот на работата користи:

```bash
git status
git diff
```

Што прават:

- `git status` - покажува кои датотеки се изменети.
- `git diff` - ги прикажува разликите пред да commit-ираш.

Ова е важно за да не внесеш промени надвор од твојата зона.

### 5. Commit на логична целина

Кога си сигурен дека промените се готови, направи commit.

```bash
git add .
git commit -m "Add login page validation"
```

Што прават:

- `git add .` - ги подготвува сите нови и изменети датотеки за commit.
- `git commit -m "..."` - креира commit со порака.

Добри commit пораки:

- `Add login validation`
- `Fix auth token refresh`
- `Update homepage layout`
- `Improve users API response`

Лоши commit пораки:

- `fix`
- `changes`
- `stuff`

### 6. Пушти ја гранката на remote

Кога commit-ите се подготвени:

```bash
git push -u origin frontend/ivan-login-page
```

Што прави:

- `git push` - ги праќа локалните commit-и на remote репото.
- `-u` - ја врзува локалната гранка со remote гранката за полесен следен push/pull.
- `origin` - името на remote репото.
- `frontend/ivan-login-page` - името на гранката.

Ако гранката веќе постои и е врзана со remote:

```bash
git push
```

## Како работи merge процесот со PM

Ова е препорачаниот процес:

1. Девелоперот работи на својата гранка.
2. Девелоперот тестира локално.
3. Девелоперот прави commit-и.
4. Девелоперот пушта на remote.
5. Девелоперот му кажува на PM дека гранката е спремна.
6. PM го прегледува кодот.
7. PM ја спојува гранката во главната гранка.
8. По merge, другите девелопери ги влечат најновите промени.

PM треба да merge-ира само кога:

- кодот компајлира или стартува без грешки
- функционалноста е тестирана
- нема ненамерни промени во погрешниот дел од репото
- промените се усогласени со задачата

## Правила за frontend тимот

Frontend тимот мора строго да се држи до `frontend/`.

### Што смеете да менувате

- `frontend/src/...`
- frontend конфигурација ако е дел од задачата
- UI, компоненти, страници, стилови, client-side логика

### Што не смеете да менувате

- `backend/app/...`
- `backend/routes/...`
- `backend/database/...`
- Laravel конфигурации
- backend dependency фајлови

### Ако frontend тимот мора да допре до backend

Ако е потребно backend API да се прилагоди за нов frontend flow:

- отворете задача со PM
- backend девелопер прави промена во backend гранка
- frontend девелопер ја прилагодува UI страната во frontend гранка
- PM ги координира двете промени

Frontend тимот не треба сам да менува backend логика "само за да функционира" без договор.

## Правила за backend тимот

Backend тимот мора строго да се држи до `backend/`.

### Што смеете да менувате

- `backend/app/...`
- `backend/routes/...`
- `backend/database/...`
- backend тестови
- Laravel конфигурации

### Што не смеете да менувате

- `frontend/src/...`
- frontend компоненти
- frontend build логика
- frontend UI одлуки

### Ако backend тимот мора да допре до frontend

Ако backend change бара frontend прилагодување:

- backend девелопер ја завршува backend промената
- frontend девелопер ја прилагодува UI страната
- PM ги обединува двете промени

Backend тимот не треба да менува frontend фајлови без договор.

## Како да избегнете мешање меѓу тимовите

1. Секој работи на своја гранка.
2. Секој ја проверува својата `git diff` пред commit.
3. Секој commit треба да биде мал и јасен.
4. Ако видиш промени во туѓа зона, сопри и прашај.
5. Ако задачата бара и backend и frontend, тоа е за координација со PM.

## Корисни Git команди за секој член на тимот

### `git branch`
Ги покажува локалните гранки.

Пример:

```bash
git branch
```

### `git switch <branch>`
Те префрла на друга гранка.

Пример:

```bash
git switch frontend/ivan-login-page
```

Ако вашата Git верзија е постара, може да се користи `git checkout <branch>`.

### `git fetch`
Ги симнува најновите промени од remote без merge.

```bash
git fetch origin
```

### `git pull`
Ги симнува и спојува промените од remote гранка во локалната гранка.

```bash
git pull origin main
```

### `git status`
Проверка на тековната состојба.

```bash
git status
```

### `git diff`
Ги прикажува деталите за промените.

```bash
git diff
```

### `git add`
Додава датотеки во следниот commit.

```bash
git add frontend/src/app/login/page.js
```

### `git commit`
Прави snapshot од промените.

```bash
git commit -m "Update login page validation"
```

### `git push`
Ги праќа commit-ите на remote.

```bash
git push
```

### `git log`
Ги прикажува последните commit-и.

```bash
git log --oneline --graph --decorate -10
```

## Препорачани работни сценарија

### Сценарио 1: Frontend девелопер работи на нова страница

1. Ги симнува последните промени со `git fetch origin`.
2. Креира нова гранка од `main`.
3. Работи само во `frontend/`.
4. Проверува со `git status` и `git diff`.
5. Прави commit-и.
6. Пушта гранка со `git push -u origin ...`.
7. Му јавува на PM.
8. PM прави merge.

### Сценарио 2: Backend девелопер поправа API проблем

1. Ги симнува последните промени со `git fetch origin`.
2. Креира backend гранка.
3. Работи само во `backend/`.
4. Тестира локално.
5. Прави commit-и.
6. Пушта гранка.
7. Му јавува на PM.
8. PM прави merge.

### Сценарио 3: Две промени зависат една од друга

1. Backend девелопер прави API промена.
2. Frontend девелопер прави UI адаптација.
3. Двајцата работат во свои гранки.
4. PM следи која промена е подготвена.
5. PM ги merge-ира во правилен редослед.

## Што да не правите

- Не работете директно на `main`.
- Не push-ирајте незавршен код без да знаете што содржи.
- Не менувајте backend ако сте frontend девелопер.
- Не менувајте frontend ако сте backend девелопер.
- Не мешајте голем број несродни промени во еден commit.
- Не игнорирајте `git diff` пред пуштање.

## Краток резиме на процесот

1. Создај гранка за твојата задача.
2. Работи само во твојата папка.
3. Проверувај `git status` и `git diff`.
4. Commit-и логично и често.
5. Push-ни ја гранката.
6. Јави му на PM.
7. PM merge-ира кога кодот е проверен.
